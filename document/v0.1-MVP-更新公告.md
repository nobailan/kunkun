## 一、版本概述

v0.1 MVP 实现了 DS-Harness 的最小可用版本：能在命令行里接任务、调工具、完成任务，跑通完整的 Agent Loop。

```
用户输入: "在 ~/project 目录下找所有 Python 文件，统计每个文件的代码行数"

Agent:
  1. ThinkBlock: DSv4 思考中 (灰色斜体实时渲染)
  2. Tool Call: glob("**/*.py") → 拿到文件列表
  3. Tool Result: 3 个文件
  4. Text Output: 统计每个文件的代码行数 → 输出结果
```


## 四、遇到的问题与解决方案

### 问题 1: Hermes 源码分析不够深入

**现象**: 在设计方案阶段，Hermes 的 clone 路径在 `/tmp/hermes-agent` (Git Bash 虚拟文件系统)，虽然文件存在，但 `Read` 工具使用相对路径时找不到。设计方案中 Hermes 相关内容主要来自 WebSearch 的间接信息。

**解决**:
1. 使用 `pwd -W` 发现 Windows 物理路径为 `C:/Windows/TEMP/hermes-agent`
2. 用完整 Windows 路径重新读取了 12 个核心源文件:
   - `run_agent.py` (275KB — 主 Agent 类)
   - `agent/conversation_loop.py` (conversation 主循环)
   - `agent/memory_manager.py` + `memory_provider.py` (可插拔记忆系统)
   - `agent/context_engine.py` (可插拔上下文引擎 ABC)
   - `agent/conversation_compression.py` (压缩编排)
   - `agent/tool_executor.py` + `tool_dispatch_helpers.py` (工具执行 + 并行策略)
   - `agent/prompt_builder.py` (System Prompt 组装)
   - `tools/registry.py` (AST 扫描自发现注册)
   - `providers/base.py` (ProviderProfile 声明式配置)
   - `hermes_state.py` (269KB — SQLite + FTS5 状态存储)
3. 将 Hermes 从 `/tmp` 迁移到 `E:\agentProject\hermes-agent\`
4. 更新设计方案 Section 1.3，从间接信息替换为源码级深度分析

### 问题 2: Windows GBK 编码导致 emoji 打印崩溃

**现象**: 验证脚本中使用 `✅` emoji 时抛出 `UnicodeEncodeError: 'gbk' codec can't encode character`

**解决**: 
- 验证脚本中避免使用 emoji，改用 `[OK]` 等纯 ASCII 标记
- 生产代码中 Rich 库自动处理 Unicode 编码，不受此影响
- 设置 `PYTHONIOENCODING=utf-8` 环境变量可彻底解决

### 问题 3: pip 安装路径选择

**现象**: 系统默认 pip 指向的 Python 环境不是 agent 开发专用环境

**解决**: 使用专用 conda 环境 `E:\anaconda3\envs\graph` 进行开发和安装
```bash
E:/anaconda3/envs/graph/python -m pip install -e E:/agentProject/DS-harness
```

### 问题 4: SSE 流格式兼容性

**现象**: DSv4 Anthropic 兼容接口的 SSE 格式可能有多种变体 (标准 SSE `event:` + `data:` 或纯 JSON 行)

**解决**: `_parse_sse_stream()` 同时兼容两种格式:
- 标准 SSE: `event: xxx \n data: xxx`
- 纯 JSON 行: 每行一个 JSON 对象
- 通过检测行前缀 (`event:` / `data:` / `{`) 自动切换解析模式
