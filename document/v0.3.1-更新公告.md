# Kun v0.3.1 更新公告

> 发布日期: 2026-07-07
> 版本: v0.3.1
> 状态: Skill 自改进闭环 — 从静态模板到可进化学习系统

---

## 一、版本概述

v0.3.1 是 v0.3 的实质性升级。核心变化：**Kun 从"使用静态 Skill 模板"进化为"拥有可自我改进的 Skill + Memory 学习系统"**。

v0.3 的 Skill 和 Memory 本质上还是静态配置——开发者写了什么就是什么。v0.3.1 借鉴 Hermes 的 `background_review` + `curator` 架构，实现了三个核心能力：

1. **每轮对话后自动反思** — 用 flash 模型评估"有什么值得记住的？""Skill 需要改进吗？"
2. **Skill 生命周期管理** — usage 追踪 + active→stale→archive 自动流转
3. **LLM 语义判断取待关键词匹配** — 记忆检索交给 LLM 而非字符串匹配

## 四、核心能力详解

### 4.1 自动反思引擎 (background_review.py)

```
每轮对话结束 (stop_reason="end_turn")
  → asyncio.create_task(review_turn())
  → flash 模型评估两个问题：
      "有什么值得记住的？" → 自动写入 MemoryManager
      "Skill 需要改进吗？" → 自动创建/修补 Skill
  → 不阻塞主对话 (fire-and-forget)
  → 会话结束 await wait_pending()
```

**与 Hermes 的差异**：Hermes 会 fork 一个完整的子 Agent（继承 runtime），用 memory + skill 工具对话。Kun 采用更轻量的方式——结构化 JSON prompt + 单次 API 调用 + 直接写入。两种方式各有优劣，Kun 的方式更省 token。

### 4.2 记忆检索：从关键词匹配到 LLM 语义判断

**v0.3 的问题**：记忆全文注入 System Prompt。用户问"帮我写 Python 代码"，Agent 却因为看到猫名记忆而顺带输出"老大叫圆宝"。

**v0.3.1 的修复**：

```
System Prompt 注入: 仅元数据索引 (name + description, 每条一行)
  ↓
Agent 看到索引 → 自行判断相关性 → 调 recall 拿全文
  ↓
recall 搜索: n-gram 中文字符匹配 + 无结果时返回全部记忆
```

三层保障：
1. **索引轻量** — 10 条记忆才 300 tokens，不到旧版全文注入的 1/10
2. **LLM 判断** — 语义理解力远超字符串匹配，"老大叫什么"不会被误解
3. **n-gram fallback** — recall 搜索时，"代码风格" 通过 3-gram 匹配到 "编码风格偏好"

### 4.3 Skill 生命周期管理

```
Skill 状态机:

  active ──(30天未加载)──▶ stale ──(90天未加载)──▶ archived
    ▲                           │
    │                           │
    └──(手动 restore)───────────┘

  pinned: 跳过自动转换
  code-review / python-project / git-conventions: 预置 Skill，永不自动归档
```

**SkillUsageStore (`skills/usage.py`)**：
- `.usage.json` 追踪每个 Skill 的 `load_count` / `patch_count` / `view_count`
- `bump_load()` — Skill 被匹配注入 System Prompt 时
- `bump_patch()` — Skill 被 background_review 修补时
- 时间戳记录 → Curator 判断是否过期

**SkillCurator (`skills/curator.py`)**：
- `scan()` — 扫描全部 Skill，自动转换生命周期
- `report()` — 返回所有 Skill 的状态报告
- `pin()` / `unpin()` — 手动固定/取消固定

### 4.4 批量记忆写入

**v0.3**：Agent 只能一条一条写。当需要"删 2 条过时的 + 改 1 条冗余的 + 加 1 条新的"时需要 4 次工具调用。

**v0.3.1**：`remember` 工具新增 `operations` 数组参数：

```json
{
  "operations": [
    {"action": "remove", "old_name": "old-pref"},
    {"action": "add", "name": "new-pref", "content": "...", "description": "..."},
    {"action": "replace", "old_name": "stale-fact", "name": "updated-fact", "content": "..."}
  ]
}
```

先校验全部操作，再执行。任一操作参数无效则全部放弃（原子性）。

## 五、和 Hermes 的对齐情况

| Hermes 核心能力 | Kun v0.3 | Kun v0.3.1 |
|----------------|----------|-----------|
| background_review 每轮反思 | ❌ | ✅ |
| Skill 使用量追踪 (usage.json) | ❌ | ✅ |
| Curator 生命周期 (active→stale→archived) | ❌ | ✅ |
| LLM 语义判断取待关键词匹配 | ❌ | ✅ |
| 批量记忆写入 (operations) | ❌ | ✅ |
| Frozen Snapshot | ❌ | ❌ |
| FTS5 全文搜索 | ❌ | ❌ |
| Provider 插件架构 | ❌ | ❌ |

**Kun 独有的优势**：
- 一个文件一条记忆 → Git diff 友好
- 元数据索引注入 → 省 token
- LLM 语义判断 → 比 FTS5 文本搜索更精准
- 零外部依赖 → SQLite/jieba 都不需要
- 显式 + 自动双路径 → 用户可主动控制，也可靠自动反思

## 七、已知限制

1. **Curator 需要手动触发** — `scan()` 没有在启动时自动运行（生命周期管理需要长期使用数据，短期内无实际效果）
2. **background_review 依赖 flash 模型** — 如果没有 flash 模型的 API 访问权限，反思会静默跳过
3. **Frozen Snapshot 未实现** — 每次 `run()` 重新 load，记忆多时可能影响 cache
4. **无 FTS5** — 跨会话历史消息搜索不可用
