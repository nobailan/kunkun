## 一、一句话定位

| 系统 | 定位 | 核心理念 |
|------|------|---------|
| **Claude Code** | 自动记忆助理 | LLM 自动提取 + 自动选择，用户无感，静态 Skill |

## 三、逐维度对比（Kunkun 状态更新）

### 3.1 存储格式

| 维度 | Claude Code | Kunkun v0.3.1 | Hermes |
|------|------------|-----------|--------|
| 记忆文件格式 | `.md` + YAML frontmatter | `.md` + YAML frontmatter | `MEMORY.md` / `USER.md`，`§` 分隔 |
| 条目粒度 | 一个文件 = 一条记忆 | 一个文件 = 一条记忆 | 一个文件 = 多条条目 |
| 索引文件 | `MEMORY.md` | `MEMORY.md` | 无 |
| 用户画像分离 | 无 | 无 | **有** — `USER.md` 独立 |
| 字符限制 | 无 | 无 | **有** — 2200/1375 chars |
| 文件锁 / 并发安全 | 无 | 无 | **有** |

### 3.2 检索方式

| 维度 | Claude Code | Kunkun v0.3.1 | Hermes |
|------|------------|-----------|--------|
| **检索方法** | **LLM 判断** — 传入 name+description，LLM 选 ≤5 条 | **元数据索引 + LLM 判断** — 所有 name+description 注入 System Prompt，Agent 自行调 `recall` 拿全文 | **FTS5 + trigram** — SQLite 全文搜索 |
| **检索成本** | 1 次额外 LLM 调用 | **零** — 复用主 LLM 调用，由 Agent 自行判断 | 零 — SQLite 查询 |
| **中文支持** | 依赖 LLM 理解 | **LLM 语义判断**（最精准）+ n-gram 字符匹配（recall fallback） | unicode61 + trigram |
| **跨会话搜索** | 不支持 | 不支持 | **支持** — FTS5 搜索全部历史消息 |
| **记忆注入策略** | 全文注入 System Prompt | **仅注入元数据索引**，Agent 自行判断是否需要全文 | Frozen Snapshot |

> **v0.3.1 关键改进**：记忆不再全文注入 System Prompt。改为注入索引（name + description），由 Agent 根据任务自行判断是否需要调 `recall` 获取全文。这解决了"问 Python 代码却蹦出猫的名字"的问题——语义判断交给了 LLM，而非关键词匹配。

### 3.3 记忆生命周期

| 维度 | Claude Code | Kunkun v0.3.1 | Hermes |
|------|------------|-----------|--------|
| **写入触发** | LLM 对话后自动提取 | **双路径**：① 用户显式说"记住"→ Agent 调 `remember` 工具 ② 每轮 background_review 自动提取 | Agent 调用 `memory` 工具 + **sync_turn** 每轮自动同步 |
| **批量写入** | 不支持 | 不支持 | **支持** — `operations` 数组 |
| **容量管理** | 无 | 无 | **有** — 字符限制 + consolidation |
| **去重** | 无 | 无 | **有** |
| **注入策略** | 动态 System Prompt | 元数据索引（每轮重新匹配） | **Frozen Snapshot** |

### 3.4 Skill 系统

| 维度 | Claude Code | Kunkun v0.3.1 | Hermes |
|------|------------|-----------|--------|
| **Skill 本质** | 静态模板（开发者写） | **可进化的工具**（用户教 + 自动反思） | **可进化的工具**（用户教 + 自动反思） |
| **每轮自动反思 → 创建/修补 Skill** | ❌ | ✅ `background_review` | ✅ `background_review` |
| **Skill 使用量追踪** (usage.json) | ❌ | ✅ `SkillUsageStore` | ✅ `skill_usage.py` |
| **Curator 生命周期管理** (active→stale→archived) | ❌ | ✅ `SkillCurator` | ✅ `curator.py` |
| **合并重叠 Skill** (consolidation) | ❌ | ❌ | ✅ |
| **跨会话学习** | 无 | ✅ 每次会话改进沉淀到下次 | ✅ |

### 3.5 架构可扩展性

| 维度 | Claude Code | Kunkun v0.3.1 | Hermes |
|------|------------|-----------|--------|
| Provider 插件 | 无 | 无 | **有** — `MemoryProvider` ABC |
| 外部 Provider | 不支持 | 不支持 | Honcho / Hindsight / Mem0 |

## 五、和 Hermes 的剩余差距

| 优先级 | Hermes 特性 | Kunkun 状态 | 差距 |
|--------|-----------|---------|------|
| 🔴 | **Skill 自改进闭环** | ✅ background_review | **已对齐** |
| 🔴 | **记忆自动提取** | ✅ background_review | **已对齐** |
| 🟡 | **Skill 使用量追踪** | ✅ SkillUsageStore | **已对齐** |
| 🟡 | **Curator 生命周期** | ✅ SkillCurator | **已对齐**（缺少 consolidation） |
| 🟡 | Frozen Snapshot | ❌ 每轮重新 load | 保护 prefix cache，中等代价 |
| 🟡 | 批量写入 (operations) | ❌ 单条写入 | 减少 LLM 往返，低代价 |
| 🟡 | FTS5 全文搜索 | ❌ n-gram + LLM 判断 | 跨会话历史搜索，中代价 |
| 🟢 | 双轨存储 (MEMORY.md + USER.md) | ❌ 统一 memory | 用户画像独立，低代价 |
| 🟢 | 容量上限 + 自我管理 | ❌ 无限制 | 防膨胀，中代价 |
| ⚪ | Provider 插件架构 | ❌ | 高代价，长期 |
| ⚪ | 外部漂移检测 + 文件锁 | ❌ | 多进程场景才需要 |

## 六、后续建议路线

### 短期（v0.4 附带）

1. **Frozen Snapshot** — 启动时加载记忆快照，会话中途不更新 System Prompt。保护 prefix cache，也是 Hermes 的重要工程决策
2. **批量写入** — `remember` 工具支持 `operations` 数组，一次调用完成 add+replace+remove

### 中期（v0.5）

3. **FTS5 全文搜索** — 引入 `aiosqlite`，建 messages 表和 FTS5 索引，支持跨会话中文搜索
4. **容量上限 + 自我管理** — 给记忆加 char limit，满时教 Agent 整理而非拒绝

### 长期（v0.6+）

5. **双轨存储** — 在现有 metadata.type 基础上独立 `USER.md` 轨道
6. **Provider 插件** — 抽象 MemoryProvider 接口，支持向量检索

## 七、总结

| | Claude Code | Kunkun v0.3.1 | Hermes |
|---|------------|-----------|--------|
| **最适合** | 日常编码，零配置 | 中文优先，可进化的编码 Agent | 企业级多后端，长期记忆 |
| **复杂度** | 低（文件系统） | 低（文件系统 + 轻量 JSON） | 高（6045 行 SQLite） |
| **Skill 进化** | ❌ 静态 | ✅ 自动反思创建/修补 | ✅ 自动反思创建/修补 |
| **Curator 管理** | ❌ | ✅ usage 追踪 + 生命周期 | ✅ 完整 curator |
| **自动化程度** | 高（LLM 自动提取） | 高（双路径：显式 + 自动反思） | 高（每轮自动 sync） |
| **中文友好** | 中（LLM 理解） | **高**（中文优先 + LLM 语义判断） | 高（trigram + jieba） |
| **Provider 扩展** | 无 | 无 | 强 |
| **生产就绪** | 是 | v0.3.1 — Skill 自改进能力已就绪 | 是 |

Kunkun v0.3.1 在**核心差异化能力**上已经超越了 Claude Code（Skill 自进化、Curator 生命周期管理），追平了 Hermes 最关键的 background_review 闭环。剩余差距主要在工程细节（Frozen Snapshot、FTS5、批量写入）和扩展性（Provider 插件）层面。
