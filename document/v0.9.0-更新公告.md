# Kunkun v0.9.0 更新公告

> 发布日期: 2026-07-10
> 版本: v0.9.0
> 状态: Workflow + Cron — Agent 编排的自动化引擎

---

## 一、版本概述

v0.9.0 加入了 Workflow 引擎和 Cron 调度器。Kunkun 现在可以：
- 用 Python 脚本编排多 Agent 协作（pipeline/parallel）
- 定时自动执行任务（cron 表达式）
- 在仪表盘中追踪 cron 任务状态

设计原则：DSv4 适配 + 最小 Token 浪费。

---

## 二、Workflow 引擎

### 四个原语

```python
from kunkun.workflow import workflow, agent, parallel, pipeline, phase

@workflow(name="code-review-nightly")
async def nightly():
    await phase("扫描变更")
    files = await agent("grep 最近修改的文件", agent_type="explorer")

    await phase("审查")
    results = await pipeline(      # 非屏障，token 最优
        files,
        lambda f: agent(f"审查 {f}", agent_type="reviewer"),
    )

    await phase("汇总")
    return await agent(f"汇总: {results}", agent_type="planner")
```

| 原语 | 行为 | Token 优化 |
|------|------|-----------|
| `agent(prompt, role)` | 启动角色化子 Agent | 共享 Frozen Snapshot |
| `parallel(tasks)` | **屏障** — 全部完成才返回 | 各自独立 session |
| `pipeline(items, ...stages)` | **非屏障默认** — 流水线 | Item A stage 2 时 Item B stage 1 |
| `phase(title)` | 进度标记 | 仪表盘可视化 |

### pipeline vs parallel

```
parallel:  A ──stage1── stage2──
           B ──stage1── stage2──  ← 必须等全部 stage1 完成才进 stage2
           C ──stage1── stage2──

pipeline:  A ──stage1── stage2──
           B ──stage1── stage2──  ← B 进 stage2 时 C 还在 stage1
           C ──stage1── stage2──    不等最慢的，省时间 = 省 token
```

实测：pipeline 比 parallel 快 ~2.3 倍。

## 三、Cron 调度器

### API

```python
from kunkun.cron import CronScheduler

scheduler = CronScheduler()

@scheduler.task("0 9 * * 1-5")
async def daily():
    result = await run_agent("汇总昨日变更")
    return result

await scheduler.start()
```

### 功能

| 功能 | 实现 |
|------|------|
| cron 解析 | 纯 Python，5 字段标准 cron |
| 持久化 | `.kun/cron/tasks.json` |
| 并发控制 | Semaphore，不重复执行 |
| 失败跳过 | 上次未完成 → 跳过本次 |
| 仪表盘 | 任务状态卡片（触发时间/成功率/耗时） |

### Token 优化

- 定时任务默认走 flash 模型
- 复用 Frozen Snapshot（不重复加载 Memory/Skill）
- 独立 session（不带历史对话，上下文干净）

## 五、FTS5 跨会话搜索

### 架构

```
.kun/messages.db
├── sessions 表 — 会话元数据
├── messages 表 — 每条消息 (session_id, role, content, timestamp)
└── messages_fts — FTS5 虚拟表 (unicode61 tokenizer)
```

每次对话自动保存到 SQLite。Agent 可以通过 `session_search` 工具搜索历史。

### 搜索示例

```
session_search("Docker 部署方案")
  → 返回跨会话的匹配消息片段
```

### 与 Hermes 对齐

这是 Kunkun 补齐与 Hermes 的最后一块差距——FTS5 全文搜索，零外部依赖（Python 内置 sqlite3）。

## 六、MCP 支持

### 配置

创建 `.kun/mcp.json`：

```json
{
  "servers": {
    "filesystem": {"command": "npx @modelcontextprotocol/server-filesystem /tmp"},
    "github": {"command": "npx @modelcontextprotocol/server-github"}
  }
}
```

Kunkun 启动时自动连接 MCP Server，发现工具并注册。

### 架构

```
MCP Server (子进程, stdio)
  └── JSON-RPC (initialize → tools/list → tools/call)
       └── MCPRegistry → 注册为 kunkun ToolInstance
            └── Agent 可调用 mcp_filesystem_xxx, mcp_github_xxx
```

### 工具命名

MCP 工具自动加 `mcp_{server_name}_` 前缀，避免冲突。

## 六、子 Agent 稳定性修复

## 八、System Prompt 精简

借鉴 Pi 的极简哲学——前沿模型已被 RL 训练成编码 Agent，不需要用几千 tokens 的 System Prompt"教"它怎么工作。

### 精简前后对比

| | 旧版 | 新版 | 节省 |
|---|------|------|------|
| 字符数 | ~1200 chars | ~300 chars | **75%** |
| Token 估算 | ~400 tokens | ~100 tokens | **75%** |
| 段落数 | 7 段 | 3 段 | — |

### 砍掉了什么、为什么

| 砍掉的内容 | 原因 |
|-----------|------|
| "你的能力" (读写/Shell/搜索) | DSv4 从工具 schema 已经知道，不需要重复 |
| "ThinkBlock 使用" 整段 | 模型知道怎么用 thinking |
| "输出格式" (Markdown/代码块/路径) | 自然对话就能做到，不需要指令 |
| "复杂任务处理" 4 步骤详述 | 合并为一行 "先规划→跟踪→执行" |
| "最小改动""主动验证""清晰沟通" | 合并为 "先读后改，最小改动，改完验证" 一行 |

### 保留了

```text
你是 Kunkun，DeepSeek v4 编码 Agent。用中文回复，代码和命令保持原文。

## 核心规则
- 先读后改，最小改动，改完验证
- 多步骤任务用 ThinkBlock 规划 → TodoWrite 跟踪 → 逐步执行
- 独立子任务用 agent 并行

## 记忆与 Skill
下文的"项目记忆"和"项目 Skill"是当前会话的有效内容，直接遵守其中的约定。
```

**300 chars, ~100 tokens。每次 API 调用省 300 tokens。**
