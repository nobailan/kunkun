## 二、Agent Loop 主循环

Kunkun 的核心是一个 `while True` 循环，借鉴 Claude Code 的 QueryEngine 设计：

```
用户输入 "分析 src/kunkun/core/ 的架构"
    │
    ▼
┌─────────────────────────────────────────┐
│  Pre-LLM                                │
│  ├── Frozen Snapshot (记忆+Skill 注入)   │
│  ├── Context Trimmer (滑动窗口裁剪)       │
│  └── Cost Router (模型路由)              │
└──────────────┬──────────────────────────┘
               ▼
┌─────────────────────────────────────────┐
│  LLM Stream (DeepSeek v4 API)           │
│  ├── reasoning_content → ThinkBlock     │
│  ├── content → text chunks              │
│  └── tool_calls → 工具调用              │
└──────────────┬──────────────────────────┘
               ▼
┌─────────────────────────────────────────┐
│  Post-LLM                               │
│  ├── Permission Check (三层门禁)         │
│  ├── Tool Execution                     │
│  └── Result → 回传 LLM (继续循环)        │
└──────────────┬──────────────────────────┘
               ▼
          stop_reason?
      ┌────┴────┐
      │         │
   end_turn  tool_use → 继续循环
      │
      ▼
┌─────────────────────────────────────────┐
│  Session End                            │
│  ├── background_review (自动记忆提取)    │
│  ├── thinking_eval (思考质量评分)        │
│  ├── task_eval (任务维度评分)            │
│  └── FTS5 auto-save (对话历史持久化)     │
└─────────────────────────────────────────┘
```

### 为什么不用 LangGraph

Agent Loop 的本质就是 `while True: call LLM → tool_use? → execute → feed back`。用 LangGraph 需要额外适配 StateGraph、Node、Edge、ConditionalEdge、Checkpointer 五个抽象层——**50 行能写完的逻辑变成 500 行框架胶水代码**。

```
                    ┌──────────────────────────┐
                    │    System Prompt 组装      │
                    └──────────┬───────────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
       ┌──────────┐    ┌──────────┐    ┌──────────────┐
       │  Memory  │    │  Skill   │    │  Context     │
       │  (事实)   │    │  (约定)   │    │  Files       │
       └────┬─────┘    └────┬─────┘    └──────────────┘
            │               │
            ▼               ▼
    .kun/memory/      skills/
    ├── MEMORY.md     ├── code-review/SKILL.md
    ├── style.md      ├── python-project/SKILL.md
    └── work.md       └── git-conventions/SKILL.md
            │               │
            ▼               ▼
    元数据索引注入     triggers匹配注入
    (Agent recall)    (System Prompt)
            │               │
            └───────┬───────┘
                    ▼
         background_review (每轮反思)
         ├── 自动提取 Memory
         └── 自动修补 Skill
                    │
                    ▼
              Curator 生命周期
              active → stale → archived
```

### Memory 和 Skill 的区别

| | Memory | Skill |
|---|--------|-------|
| 本质 | "是什么"（事实/偏好） | "怎么做"（约定/规范） |
| 注入方式 | 元数据索引 → Agent 自行 recall | 全文注入 System Prompt |
| 写入 | 用户说"记住" + background_review | background_review 自动修补 |
| 生命周期 | 手动管理 | Curator 自动管理 |

## 四、评测体系

Kunkun 是唯一内置评测的编码 Agent。三个评测模块来自三篇论文：

### ThinkBlock 评测（AgentThink 论文）

```
DSv4 reasoning_content 提取
  → flash 模型三维度评分:
      ├── 分析瘫痪 (Analysis Paralysis)
      ├── 流氓操作 (Rogue Actions)
      └── 过早放弃 (Premature Disengagement)
  → 评分报告 + 日志
```

### AdaRubric 任务评测（AdaRubric 论文）

```
任务 prompt + 执行轨迹
  → flash 模型动态生成评分维度
  → 每个维度 0-3 分
  → 例如: "搜索完整性: 2/3, 结果准确性: 3/3"
```

### GRPO 多版本生成（GRPO 范式）

```
同一任务
  → 3 条路径并行 (direct / search_first / design_first)
  → flash 模型 LLM-as-Judge 择优
  → 返回最优结果 + 选择理由
```

## 六、技术特点

| 特性 | 实现 |
|------|------|
| **零框架** | 不用 LangChain/LangGraph，自研 Agent Loop |
| **DSv4 专属** | ThinkBlock 解析、Prompt 粒度编译器、MLA 感知上下文 |
| **评测内置** | 3 种评测模式，来自 3 篇论文的 prompt 落地 |
| **Frozen Snapshot** | 会话首次构建 System Prompt，后续复用（保护 prefix cache） |
| **FTS5 全文搜索** | SQLite + unicode61，跨会话历史搜索 |
| **Workflow + Cron** | Python 脚本引擎 + asyncio 调度器 |
| **MCP 扩展** | stdio + JSON-RPC，连接外部 MCP Server |
