## 审查目标

对代码进行全面审查，发现潜在问题并给出可操作的改进建议。

## 审查维度

### 1. 命名规范
- 类名大驼峰 (PascalCase)、函数/变量名小驼峰 (camelCase) 或蛇形 (snake_case)
- 常量全大写 + 下划线
- 命名应自解释，避免单字母 (循环变量除外)

### 2. 逻辑正确性
- 边界条件处理 (空列表、None、零值)
- 异常处理是否完善 (不用空 catch，异常信息要具体)
- 资源是否正确释放 (文件句柄、网络连接)

### 3. 安全性
- SQL 注入风险 (参数化查询)
- 路径遍历风险 (用户输入拼接文件路径)
- 敏感信息是否硬编码 (API Key、密码)

### 4. 可维护性
- 单函数不超过 50 行、单类不超过 300 行
- 圈复杂度不超过 10
- 公开方法必须有 docstring
- 重复代码应抽取为公共函数

### 5. 性能
- 循环内是否避免了不必要的重复计算
- 大数据量操作是否考虑了分页/流式处理
- 是否有 N+1 查询问题

## 输出格式

使用表格呈现审查结果：

| 严重度 | 文件:行号 | 问题 | 建议修复 |
|--------|----------|------|---------|
| 🔴 严重 | | | |
| 🟡 建议 | | | |
| 🟢 优化 | | | |

**严重度定义**:
- 🔴 严重: 会导致 bug、安全漏洞或数据丢失
- 🟡 建议: 不符合规范、可能引发问题
- 🟢 优化: 更好的写法、性能提升
